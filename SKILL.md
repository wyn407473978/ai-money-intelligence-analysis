# AI赚钱情报分析 Skill

## 1. Skill 名称和用途

**Skill 名称**: ai-money-intelligence-analysis

**用途**: 分析 AI 新闻/热点采集程序输出的 JSON 数据，从中筛选出适合「AI赚钱情报站」使用的内容选题、商业机会、短视频爆款方向和风险提醒。

**适用场景**: 每次运行完 `ai_news_fetcher.py` 采集脚本后，需要对输出的 JSON 数据进行二次分析，产出面向普通人的赚钱情报报告。

---

## 2. 什么时候使用这个 Skill

当以下任一条件满足时，调用本 Skill：

- `ai_news_fetcher.py` 运行完毕，JSON 数据已获取
- 用户说"帮我分析今天的 AI 热点"或类似指令
- 需要为「AI赚钱情报站」生成日报/周报

---

## 3. 输入数据要求

### 必须提供

`ai_news_fetcher.py` 输出的完整 JSON，包含以下字段：

```json
{
  "generated_at": "...",      // 报告生成时间
  "items": [                  // 聚合后的热点列表（重点分析这里）
    {
      "cluster_key": "...",    // 热点唯一标识，用于去重
      "title": "...",
      "hot_score": 545,
      "mention_count": 7,
      "source_count": 7,
      "sources": ["WSJ", "CNBC"],
      "content_tier": "content_ready",
      "content_category": "breakout_news",
      "content_angle": "...",
      "suggested_hook": "...",
      "money_score": 90,
      "money_grade": "S",
      "money_track": "AI自动化赚钱",
      "money_angle": "...",
      "monetization_ideas": ["..."],
      "money_reasons": ["..."],
      "risk_notes": [],
      "articles": [...]
    }
  ],
  "errors": [],               // 忽略，不影响分析
  "raw_items": []            // 仅当需要细节时查看
}
```

### 如果是原始数据（未聚合）

当 `aggregate: false` 时，数据在 `raw_items` 里而不是 `items`。此时：

1. 先按 `source` 分组
2. 同一主题的多篇报道视为一个 cluster
3. 取 `hot_score` 最高的文章作为代表进行分析

---

## 4. 分析流程

按以下顺序执行，每一步都要完成才能进入下一步：

### Step 1: 读取 items，检查是否为空

```
IF items 为空或 length == 0:
    输出"今日无数据，可能是所有源都失败了"
    停止分析
```

### Step 2: 按优先级对 items 排序

排序字段优先级（从高到低）：

1. `money_grade`（S > A > B > C）
2. `money_score`（数值大的在前）
3. `hot_score`（数值大的在前）
4. `mention_count`（数值大的在前）

```
排序后，取前 15 条进入详细分析
如果总数不足 15 条，全部分析
```

### Step 3: 过滤噪音（见第六节噪音过滤规则）

在分析之前，先把明显噪音的 items 标记为 C 并排除。

### Step 4: 对每条 item 重新判断等级

不要完全相信原始 `money_grade`，按第七节规则二次判断。

### Step 5: 生成报告

按第八节输出报告模板组织内容。

---

## 5. 评分和分级规则

### 5.1 分数体系理解

| 字段 | 含义 | 使用方式 |
|------|------|----------|
| `hot_score` | 热点强度，被多个来源报道则高 | 判断是否真的在传播 |
| `money_score` | 赚钱价值，0-100 | 参考值，不迷信 |
| `mention_count` | 同热点被多少篇文章提到 | 越高越说明是持续热点 |
| `source_count` | 不同来源数量 | 越高说明传播面越广 |

### 5.2 强制降级规则（硬规则，必须优先于所有其他判断）

以下规则优先级最高，任何一条触发即强制降级，Agent 不得绕过。

#### 规则 1：单一来源硬上限

如果一个热点同时满足：
- mention_count = 1
- source_count = 1

则该热点**最高只能评 A**。

除非同时满足以下所有条件（缺一不可）：
- 来源是顶级权威源或官方源：OpenAI、Google、Anthropic、DeepSeek、Apple、Microsoft、NVIDIA、字节、阿里、腾讯、华为、Reuters、WSJ、Bloomberg、FT
- 事件类型属于：重大产品发布（如新模型发布）、重大政策（如政府 AI 监管）、重大安全事故（如 AI 导致的事故）、重大平台规则变化
- 有明确普通人参与路径或明确行业付费场景

**如果不满足全部 3 个条件，最高只能评 B。**

#### 规则 2：官方稿限制

如果来源是公司官方发布（公司官网、官方博客、官方新闻稿），且没有其他媒体或社区跟进：
- 不得评为 S
- 应标记为"待验证"
- 最多评 A
- 如果普通人无法直接参与或缺少具体落地案例，降为 B

#### 规则 3：技术内容限制

如果内容类型属于以下任一，且没有明确应用场景、用户需求或赚钱路径：
- paper / technical paper
- benchmark / 性能测试
- inference / latency / 推理优化
- model architecture / 模型架构
- technical report / 技术报告

则**最高只能评 B**，只能写成"技术趋势观察"，不能写成"赚钱机会"。

#### 规则 4：医疗/金融/法律高风险类别

如果内容涉及以下领域，即使热度很高，也必须：
- 增加explicit风险提醒
- 标题和钩子不得暗示违法、违规、诊断、投资收益或替代专业人士

| 类别 | 禁止表达 | 正确表达 |
|------|----------|----------|
| 医疗 | "AI 可以给病人看病"、"AI 医生来了"、"医生要失业了" | "AI 做成医生助手了"、"AI 医疗要落地了吗？"、"医生工作流会被 AI 改造吗？" |
| 金融 | "AI 预测股价"、"AI 帮你投资赚钱"、"AI 替代基金经理" | "AI 帮助分析财务数据"、"AI 在金融行业的应用趋势" |
| 法律 | "AI 打官司"、"AI 替代律师"、"AI 判决" | "AI 帮助整理法律文档"、"AI 在法律行业的应用趋势" |
| 投资 | "AI 带你赚钱"、"AI 投资组合"、"AI 炒币" | "AI 辅助投资决策（不构成投资建议）" |

---

### 5.3 二次分级判断标准（在强制降级规则之后应用）

#### S 级机会高门槛规则（硬规则，优先于所有二次判断）

以下条件**必须同时满足**才能评为 S 级，任何一条不满足则最多只升到 A+：

1. **evidence_level = strong**（多源报道 mention_count >= 3，source_count >= 3，权威源，hot_score 高）
2. **publish_now = yes**（证据强、可信度高、风险可控、有明确内容方向）
3. **mention_count >= 4 或 source_count >= 4**（二选一，满足即达标）
4. **不属于医疗/金融/法律/投资等高合规赛道**

**高合规赛道特殊规则（硬规则）**：

如果内容涉及医疗、金融、法律、投资等高合规行业，即使热度高、传播广，**默认最高只能评 A+**，不得评为 S。

**例外**：只有同时满足以下全部条件，才可突破 A+ 上限评为 S：
- mention_count >= 6 且 source_count >= 6
- 有极强公共传播（引发大众媒体广泛讨论，非垂直行业内部传播）
- 有明确、低门槛的普通人参与路径
- 无需专业资质即可参与

#### S 级判断（同时满足以上高门槛规则）

- [ ] hot_score >= 200
- [ ] money_score >= 70
- [ ] mention_count >= 3
- [ ] content_tier == "content_ready"
- [ ] 有明确普通人参与路径
- [ ] 通过强制降级规则（未被降级）
- [ ] **同时通过 S 级高门槛规则（第1-4条全部满足）**

如果同时满足前 6 条，且第 7 条高门槛全部通过，判为 S。

#### A 级判断（满足任意 3 条，且未被强制降级）

- hot_score >= 150
- money_score >= 60
- mention_count >= 2
- content_tier == "content_ready"
- 有较明确的参与路径但需要解释

#### B 级判断（满足任意 2 条，或被强制降级至 B）

- hot_score >= 100
- money_score >= 40
- content_tier == "needs_processing"
- 技术趋势或行业背景，但不适合直接起号
- 被强制降级规则命中

#### C 级判断（满足任意 1 条或以下情况）

- hot_score < 100
- money_score < 40
- `content_tier` == "noise"
- 以下噪音类型（见噪音过滤规则）

### 5.4 证据等级、可信度、发布建议

每条 S/A 级机会必须输出以下三个字段：

#### evidence_level（证据等级）

| 取值 | 含义 | 判断条件 |
|------|------|----------|
| strong | 强证据 | 多源报道（mention_count >= 3，source_count >= 3）+ 权威源 + hot_score 高 |
| medium | 中等证据 | 权威源单点报道，或多源报道但热度一般（mention_count 2-3） |
| weak | 弱证据 | 单一来源、官方稿、传闻、缺少社区讨论 |

#### confidence（可信度）

| 取值 | 含义 | 判断条件 |
|------|------|----------|
| high | 高可信 | 事实清楚，来源可靠，传播证据强，mention_count >= 3 |
| medium | 中可信 | 事实基本清楚，但传播或落地路径还需观察 |
| low | 低可信 | 单一来源或信息不完整，需要人工核查 |

#### publish_now（是否建议今天发布）

| 取值 | 含义 | 判断条件 |
|------|------|----------|
| yes | 今天可以发 | 证据强、可信度高、风险可控、有明确内容方向 |
| wait | 等更多信息 | 单一来源、官方稿、缺少落地路径、需要验证 |
| no | 不建议发 | 风险高（如医疗诊断）、噪音内容、证据太弱 |

#### 5.5 机会类型判断规则

每条 S/A 级机会必须标注「机会类型」和「主类型」。

##### 机会类型定义

| 类型 | 特点 | 典型场景 |
|------|------|----------|
| **流量型** | 适合涨粉、蹭热点、做短视频/图文，不一定马上能卖东西 | DeepSeek 新模型、OpenAI 大动作、AI 伦理争议 |
| **产品型** | 适合做资料包、模板包、工具清单、课程、付费社群 | AI办公自动化、AI短视频工作流、AI提示词模板 |
| **服务型** | 适合接单、咨询、企业交付、代运营、自动化搭建 | AI客服、AI获客、企业知识库、垂直行业解决方案 |
| **观察型** | 适合做周报、深度分析、趋势储备，不适合今天强推变现 | 政策监管、技术论文、单一来源行业案例 |

##### 主类型判断

一条机会可以有多个类型，但必须标出**主类型**（最适合今天做的类型）。

判断主类型优先级：
1. 如果 `publish_now = yes` → 主类型通常是**流量型**
2. 如果有明确 toC 产品化路径 → **产品型**
3. 如果有明确 toB 交付路径 → **服务型**
4. 如果 `publish_now = wait` 或 `= no` → **观察型**

##### 输出格式

```
- 机会类型：流量型 + 产品型
- 主类型：流量型
```

---

### 5.6 发布决策一致性规则（硬规则）

#### publish_now = wait 的选题限制

如果某条机会的 `publish_now = wait`，**原则上不能进入**「今日最适合发布的 3 个选题」。

**例外**：如果当天 `publish_now = yes` 的机会不足 3 条，可以放入 `wait` 类型机会，但必须**明确标注**：

```
- 选题类型：观察型选题
- 是否建议今天强推：否
- 为什么放进来：今天 publish_now=yes 机会不足 3 条，它适合做趋势储备
```

**禁止**：把 `publish_now=wait` 的机会包装成"今天必须发"、"马上可以赚钱"。

#### 输出时必须标注

```markdown
### 选题 3：日本 AI 监管趋势观察

- 选题类型：观察型
- 对应机会等级：A
- 是否建议今天强推：否
- 发布建议：不要用强情绪标题，适合放进周报或深度内容
```

#### publish_now = no 的绝对禁止

如果 `publish_now = no`，**绝对不得**进入「今日最适合发布的 3 个选题」。这类内容今天不应该发布。

### 5.7 爆款钩子事实约束（硬规则）

#### 禁止在钩子里写的硬事实

爆款钩子可以有冲突、有情绪，但**绝对禁止**包含原始数据没有证实的硬事实：

| 禁止在钩子里写 | 原因 |
|----------------|------|
| "比 GPT-4 更强" | 除非数据有明确 benchmark 对比 |
| "比 GPT-4 便宜几十倍" | 除非数据明确包含价格信息 |
| "已经开源" | 除非数据明确说"开源"或"open source" |
| "普通人已经可以直接使用" | 除非数据明确说"开放 API"或"公开可用" |
| "替代了多少人" | 除非数据是经过核实的具体案例 |
| "某公司已经裁员" | 除非数据明确报道 |
| "医生要失业" | 绝对禁止，是医疗违规表达 |
| "AI 可以诊断" | 绝对禁止，是医疗违规表达 |
| "某产品一定能赚钱" | 未经证实 |

#### 正确钩子表达示例

| 场景 | 不推荐（违规） | 推荐（合规） |
|------|--------------|-------------|
| DeepSeek V4 | "DeepSeek V4 比 GPT-4 便宜几十倍，普通人怎么用它赚钱？" | "DeepSeek V4 据报道发布预览版，普通人该提前关注哪些机会？" |
| OpenAI 医疗 | "OpenAI 把 ChatGPT 送进医院，医生要失业了"、"医生版 ChatGPT"、"AI 可以看病" | "OpenAI 推出面向临床医生的 ChatGPT"、"医生工作流助手"、"AI 医疗真的要落地了吗？" |
| 医疗产品通用 | "AI 版 XX 来了"、"AI 替代 XX" | "面向 XX 的 AI 助手"、"AI 帮 XX 做 XX" |

#### 医疗/金融/法律产品中文表述规范（硬规则）

所有涉及医疗、金融、法律、投资等高合规赛道的产品，在中文报告中的表述必须遵循以下规则：

| 产品类型 | 禁止表述 | 正确表述 |
|----------|----------|----------|
| 医疗 AI 产品 | "医生版 ChatGPT"、"AI 看病"、"AI 诊断"、"AI 替代医生" | "面向临床医生的 ChatGPT"、"医生工作流助手"、"AI 辅助医生整理病历/文书" |
| 金融 AI 产品 | "AI 预测股价"、"AI 帮你理财"、"AI 替代基金经理" | "面向金融从业者的 AI 助手"、"AI 辅助财务数据分析" |
| 法律 AI 产品 | "AI 打官司"、"AI 替代律师"、"AI 判决" | "面向律师的 AI 助手"、"AI 辅助法律文档整理" |
| 投资 AI 产品 | "AI 带你赚钱"、"AI 投资组合"、"AI 炒币" | "AI 辅助投资决策（不构成投资建议）" |
| AI Agent | "130 个工程师被替代了" | "一家外卖公司的 AI Agent 曝光了，它到底能做什么？" |
| 新模型发布 | "已经开源，普通人直接用" | "据报道新模型来了，但目前是预览版，正式版还要等多久？" |
| 价格对比 | "比 GPT-4 便宜 50 倍" | "据报道成本大幅降低，这会改变 AI 定价格局吗？" |

#### 保守钩子模板库

当数据不完整时，使用以下模板：

- "据报道……，普通人该提前关注什么？"
- "……预览版来了，这会不会改变……？"
- "如果后续……，机会在哪里？"
- "……值得观察，但还需验证"
- "……是信号还是炒作？"

### 5.8 分级结果可能与原始 grade 不同

这是正常的。程序给的是机器判断，人的分析要结合上下文二次校准。

例如：程序给了 `money_grade: A`，但你分析后发现热点极强、普通人参与路径明确，可以升为 S。

反过来也一样。

## 6. 保守表达与事实核查规则

### 6.1 禁止的夸大表达

以下表达**绝对禁止**，除非原始数据中明确包含这些信息：

| 禁止表达 | 原因 |
|----------|------|
| "已经开源" | 除非数据明确说"开源"或"open source" |
| "可以直接使用" | 除非数据明确说"开放 API"或"公开可用" |
| "比 ChatGPT 更强" | 除非数据有明确的 benchmark 对比 |
| "替代了 X 个人" | 除非数据是经过核实的具体案例 |
| "全网爆了" | 除非 mention_count >= 5 且 source_count >= 4 |
| "正在刷屏" | 除非有明确的社交媒体传播证据 |
| "所有人都在讨论" | 夸大传播范围 |
| "今天最强热点" | 如果 mention_count >= 5 且 source_count >= 4 才可以用 |
| "唯一 S 级" | 不得在结论中使用"唯一"等绝对化表达 |

### 6.2 保守表达对照表

当数据没有明确确认时，使用以下表达：

| 场景 | 不能写 | 应该写 |
|------|--------|--------|
| 数据只说"发布新模型" | "已经开源，普通人可以直接使用" | "据报道发布了新模型，如果后续开放 API 或开源，普通人可以..." |
| 数据只说"预览版" | "马上可以用了" | "目前是预览版，正式版时间待定" |
| 数据只说"官方声称" | "效果真的很好" | "据官方声称，效果提升明显，但还需第三方验证" |
| 数据只有单一来源 | "这是今天最强热点" | "这是值得观察的单点信号，需要更多媒体确认" |
| 传播证据较弱 | "全网都在讨论" | "目前传播证据较弱，mention_count=1" |
| 不确定能否落地 | "马上可以赚钱" | "如果后续开放 API 或有成功案例，可以考虑做内容" |

### 6.3 单一媒体深度报道标注规则（硬规则）

如果某个**技术参数、融资信息、内部消息、人事变动**来自单一媒体的深度报道（如36氪、晚点、机器之心等垂直行业媒体），而非多源确认的事实，必须在报告中**明确标注来源**。

#### 标注要求

```markdown
- 据[媒体名称]报道：[信息内容]（需多方验证）
```

#### 必须标注的场景

| 内容类型 | 示例 | 标注方式 |
|----------|------|----------|
| 技术参数（参数量、上下文窗口等） | "V4 最大参数量 1.6T" | "据36氪引用知情人士说法，V4 最大参数量约 1.6T（需多方验证）" |
| 融资信息 | "DeepSeek 正在融资" | "据晚点报道，DeepSeek 目前正在进行新一轮融资（需确认）" |
| 内部消息/人事变动 | "梁文锋否决了某方案" | "据36氪报道，DeepSeek 内部曾就某方向产生分歧（单一来源，需验证）" |
| 产品策略 | "DeepSeek 组建了数十人产品团队" | "据36氪引用知情人士，DeepSeek 已组建产品团队探索 Agent 方向（需验证）" |

#### 禁止的行为

- 不能把单一媒体报道当作多源确认的事实使用
- 不能在标题或钩子中引用单一来源的内部消息作为核心事实
- 不能暗示单一来源的信息是行业共识

#### 何时可以不使用标注

- 多源（mention_count >= 3 或 source_count >= 3）确认的事实
- 官方公开声明、官方账号发布的内容
- 顶级权威媒体（Reuters、WSJ、Bloomberg、FT）的报道（但仍属于引用，需注明来源）

### 6.4 来源引用规范

**禁止使用**的表达：
- "搬运某媒体观点"
- "抄袭某文章内容"
- "直接复制"

**应该使用**的表达：
- "参考公开报道并二次整理"
- "基于 [来源名] 的报道做自己的分析框架"
- "引用来源时需要注明出处"

## 7. 噪音过滤规则（对应原第六节）

### 7.1 直接排除以下类型（判为 C，不进入报告）

| 类型 | 排除原因 | 判断方法 |
|------|----------|----------|
| Product Hunt 普通产品发布 | 缺少传播和争议，普通人无法参与 | `source` 包含 "Product Hunt" 且无高 hot_score |
| 纯技术 benchmark 论文 | 普通用户感知弱，无情绪点 | `content_category` == "technical_context" 且无落地场景 |
| AI 只是顺带出现的新闻 | 标题或内容中 AI 不是核心 | `content_tier` == "noise" |
| 纯会议/纯公告/纯招聘 | 无实际内容价值 | `content_category` 包含 "conference" 或 "announcement" |
| 深度行业文章（36氪、晚点等） | 有价值但需要二次加工，当前时间窗口不适合直接搬运 | source 包含 "36Kr"、"晚点" 且 content_category 为 breakout_news 或 breakout_tech，且与热点同期 |
| 没有传播量的小工具 | 自嗨型产品 | `hot_score` < 50 且 `mention_count` == 1 |
| 市政会议/政策中顺带提 AI | AI 只是背景词 | 标题不含 AI 核心关键词 |
| AI 生成色情/犯罪内容相关 | 违规风险高，不适合做内容 | `content_category` == "ai_generated_crime" 或风险提示含"色情/犯罪" |
| 医疗 AI 但无合规说明 | 医疗内容高风险 | `content_category` == "healthcare_ai" 且 risk_notes 为空且无"合规"/"科普"角度 |

### 7.2 降权处理（可以进入报告但降低优先级）

以下情况保留但排在后面：

- 纯技术源（OpenAI Blog/Google DeepMind/HuggingFace）为主
- 标题过于专业，普通人看不懂
- 变现路径不明确但有长期趋势价值

## 8. 输出报告模板（对应原第七节）

报告必须使用以下 Markdown 格式，**按顺序全部输出**，不可跳过任何章节（如果某章节为空，明确写"本节为空"）。

### 7.1 报告头部

```markdown
# AI赚钱情报日报

**生成时间**: {generated_at}
**数据范围**: 最近 {hours} 小时
**原始数据**: {total_count} 条原始文章 → 聚合为 {count} 条热点
```

### 7.2 第一节：今日结论

用 3-5 条 bullet 总结今天最重要的判断。

格式：
```
- **判断结论**：原因（包含具体数据依据）
```

**写法要求**：
- 每条 bullet 包含一个核心结论 + 原因
- 原因里要引用具体数字，如 `hot_score=545`、`source_count=7`
- 不要写"今天热点很多"这种废话，要具体

**示例**：
```
- 今天最值得做的是 **DeepSeek 新模型**，因为 hot_score=545、mention_count=7，多家媒体同时报道，中美AI竞争情绪强，适合做短视频和图文。
- 医疗 AI 方向 money_score=85 但 risk_notes 为空，说明合规风险未评估，不适合直接做诊断类内容，建议降权处理。
- 今天技术源（HN/HuggingFace）内容偏多，但缺少普通人能参与的落地场景，适合作为背景资料，不适合起号。
```

### 7.3 第二节：S级机会

**数量限制**：S 级机会最多 2 条。宁愿少写，也不要把弱机会写强。

**格式要求**：每条机会必须包含**所有**以下字段，完整填写，不可省略。

```markdown
### 机会 N：{标题}

- **等级**: S
- **证据等级**: strong / medium / weak（含义见 5.4 节）
- **可信度**: high / medium / low（含义见 5.4 节）
- **是否建议今天发布**: yes / wait / no（含义见 5.4 节）
- **机会类型**: 流量型 / 产品型 / 服务型 / 观察型（含义见 5.5 节，可写多个但必须标主类型）
- **主类型**: {最重要的类型}
- **方向**: {money_track 原文，如"AI自动化赚钱"}
- **热度依据**: hot_score={数值}，mention_count={数值}，source_count={数值}，来源：{sources 列表}
- **为什么值得做**: {3-5 句话，说明商业价值和传播逻辑}
- **需要核实的信息**: {列出数据中没有确认的内容}

**商业机会拆解**：
- **谁会付钱**: {具体人群}
- **付钱买什么**: {具体产品/服务}
- **交付物**: {具体是什么}
- **第一单路径**: {具体路径}
- **启动成本**: 低 / 中 / 高
- **变现难度**: 低 / 中 / 高（DeepSeek 类热点默认不低于"中"，因为内容竞争激烈；除非有非常具体、低门槛、当天可交付的产品，才可填"低"）
- **变现周期**: 当天 / 1周内 / 1个月内 / 长期
- **更适合谁**: 个人 / 小团队 / 企业
- **推荐承接方式**: {例如：免费短视频 → 私域 → 资料包}

- **爆款钩子**: {1-2 句话，有情绪点或认知冲击，但不得含未经证实的硬事实（见 5.7 节）}
- **适合平台**: {抖音/小红书/B站/知乎/公众号，选 1-2 个最合适的}
- **目标人群**: {具体描述}
- **普通人参与方式**: {具体步骤}
- **风险提醒**: {如果有风险写具体内容，如果没有写"暂无明显风险"}
- **建议动作**: {今天能做的具体行动}
```

### 7.4 第三节：A级机会

**数量限制**：A级机会最多 3 条。

格式同 S 级，但可以精简。每条必须包含：

- 等级
- 证据等级
- 可信度
- 是否建议今天发布
- 机会类型
- 主类型
- 方向
- 热度依据
- 为什么值得做
- 需要核实的信息
- 商业机会拆解（格式同 S 级）
- 爆款钩子
- 适合平台
- 目标人群
- 普通人参与方式
- 风险提醒

如果本节为空，写：
```
本节为空。今日无明显A级机会。
```

### 7.5 第四节：B级观察

**数量限制**：B 级观察最多 3 条。

只保留有长期价值的趋势。格式：

```markdown
### 观察 N：{标题}

- **观察点**: {描述正在发生的趋势}
- **为什么现在不能直接做**: {解释障碍}
- **后续应观察的信号**: {什么情况下这个方向会变成A级或S级}
```

如果本节为空，写：
```
本节为空。今日无明显B级观察。
```

### 7.6 第五节：今日不适合做的内容

列出被过滤或降权的内容类型，说明原因。

格式：
```
### 噪音类型 N：{类型名称}

**原因**: {具体说明}
**代表案例**: {如果有的话，写标题或 cluster_key}
```

如果本节为空（即所有内容都有价值），写：
```
本节为空。今日内容质量较高，无明显噪音。
```

### 7.7 第六节：今日最适合发布的 3 个选题

**数量固定**：必须恰好 3 个选题，不要多也不要少。

**这是报告最重要的部分**，必须给出 3 个具体可执行的选题。

```markdown
### 选题 1：{标题}

- **选题类型**：主攻 / 辅助 / 观察（含义见下方）
- **对应机会等级**：S / A / B
- **是否建议今天强推**：是 / 否
- **开头3秒钩子**: {短视频/图文开头第一句话，要有情绪或悬念}
- **内容结构**: {正文怎么展开，3-5 个要点}
- **结尾引导**: {引导互动的方式，但避免"割韭菜"式营销腔调，见下方风格规范}
- **适合平台**: {具体平台}
- **变现承接方式**: {怎么变现，如"引导私域卖课"、"挂载小清单"等}
- **入选原因**: {为什么选这个选题，简要说明}
```

**如果是观察型选题**，必须额外写：

```
- **是否建议今天强推**：否
- **发布建议**：不要用强情绪标题，适合放进周报或深度内容
```

#### 结尾引导风格规范（硬规则）

结尾引导要像**专业情报站**，不要像**割韭菜营销号**。

| 禁止的结尾风格 | 正确示例 |
|---------------|---------|
| "关注我，下期讲怎么用XX赚钱" | "后续会持续跟踪这个方向的进展，有新进展时会同步" |
| "今天必须做，不做就晚了" | "这个方向值得关注，可以先保持观察" |
| "想进的评论区扣1" | "对这个方向有兴趣的，可以进一步交流" |
| "限量报名"、"马上关闭" | "相关资料和进展整理好了，有需要的可以了解" |
| "普通人也能躺赚" | "这类机会需要一定的信息整合能力，不适合所有人" |
| "今天不发就亏了" | "今日窗口期值得关注，但不是非发不可" |

**原则**：
- 结尾引导以**信息价值**为中心，而非以**催促行动**为中心
- 可以引导互动，但不要制造焦虑感或紧迫感
- 可以引导关注，但语气要像"值得关注的同行"，不要像"想赚你钱的微商"

#### 选题类型定义

| 类型 | 定义 | 是否建议今天强推 |
|------|------|-----------------|
| **主攻** | publish_now=yes，证据强，适合快速涨粉/引流的选题 | 是 |
| **辅助** | publish_now=yes 或 medium，垂直行业机会，适合建立专业度 | 适度 |
| **观察** | publish_now=wait 或 publish_now=no，适合做趋势储备，不强推变现 | 否 |

#### 选题选取标准

1. 从 S 级和 A 级机会中选择
2. 优先选有情绪爆点的
3. 优先选普通人能马上跟做的
4. 优先选适合短视频传播的
5. 如果 publish_now=yes 不足 3 条，缺的配额用观察型补足，但必须明确标注

#### 禁止

- 禁止把 `publish_now=no` 的内容放进 3 个选题
- 禁止把 `publish_now=wait` 包装成"今天必须发"
- 禁止凑数——如果实在没有 3 个合适的，写 2 个也行，但不要把弱机会写强

### 7.8 第七节：今日选题组合策略

**必须输出**，说明这 3 个选题为什么这样搭配。

```markdown
## 今日选题组合策略

- **主攻选题**: {选题1标题}
- **辅助选题**: {选题2标题}
- **储备选题**: {选题3标题（如果有观察型）}

- **为什么这样搭配**:
  {3-5 句话，说明选题之间的逻辑关系和协同效应}

- **今天不建议碰的方向**:
  {1-3 句话，说明今天不应该做什么}
```

#### 搭配逻辑参考

- 主攻选题负责流量（短视频/图文，蹭热点）
- 辅助选题负责建立专业度（垂直人群，长期价值）
- 储备选题负责趋势布局（周报/深度分析用，今天不强推）

#### 搭配示例

```
- 主攻选题：DeepSeek V4 预览版（流量型，负责涨粉）
- 辅助选题：OpenAI 医疗 AI（垂直型，建立专业度）
- 储备选题：日本 AI 金融监管（观察型，趋势储备）

为什么这样搭配：
  DeepSeek 负责拉流量，是今天最强热点；
  OpenAI 医疗负责垂直人群，医生/医疗从业者有高客单价价值；
  日本监管负责建立专业度，同时告诉读者"今天不强推这个，先储备"。

今天不建议碰的方向：
  单一来源企业 AI Agent 官方稿、未经核实的性能对比、
  医疗诊断类内容。
```


## 9. 高质量分析示例（对应原第八节）

### 9.1 示例一：DeepSeek 多源报道（可评 S 级）

```json
{
  "cluster_key": "deepseek_new_model",
  "hot_score": 545,
  "mention_count": 7,
  "source_count": 7,
  "sources": ["WSJ", "CNBC", "Bloomberg", "36Kr"],
  "content_tier": "content_ready",
  "money_score": 90,
  "money_grade": "S"
}
```

**判断**：mention_count=7，source_count=7，多源权威报道，可评 S。

**保守表达示例**：
- ✅ "DeepSeek V4 据报道已发布，hot_score=545，7 家媒体报道，是今日传播最强的热点"
- ✅ "适合马上做内容"
- ❌ "已经开源，普通人可以直接使用"（数据未明确支持）
- ❌ "比 ChatGPT 更强"（数据无 benchmark 对比）

### 9.2 示例二：OpenAI 医疗产品（可评 A/S 级，需加风险）

```json
{
  "cluster_key": "openai_chatgpt_for_clinicians",
  "hot_score": 390,
  "mention_count": 4,
  "source_count": 4,
  "sources": ["MobiHealthNews", "OpenAI"],
  "content_tier": "content_ready",
  "money_score": 90,
  "risk_notes": ["医疗内容不能做诊断承诺"]
}
```

**判断**：mention_count=4，source_count=4，多源报道，但 OpenAI 官方来源为主。风险提醒已明确。

**表达示例**：
- ✅ "OpenAI 据报道推出面向临床医生的 ChatGPT 版本"
- ✅ "AI 医疗真的要落地了吗？医生工作流会被 AI 改造吗？"
- ❌ "ChatGPT 可以给病人看病了"（违规）
- ❌ "AI 医生来了，医生要失业了"（违规）

### 9.3 示例三：单一公司 AI Agent 官方稿（降级至 B）

```json
{
  "cluster_key": "delivery_hero_herogen",
  "hot_score": 182,
  "mention_count": 1,
  "source_count": 1,
  "sources": ["Delivery Hero"],
  "content_tier": "needs_processing",
  "money_score": 84,
  "money_grade": "S"
}
```

**按强制降级规则判断**：
- mention_count=1，source_count=1 → 触发单一来源硬上限
- 来源是 Delivery Hero 官方稿 → 触发官方稿限制
- 普通人不能直接使用这个企业级产品 → 缺少参与路径

**结果**：降级至 B，不能评 S。

**正确表达**：
- 等级：B
- 证据等级：weak
- 可信度：low
- 是否建议今天发布：wait
- ✅ "一家外卖公司发布企业级 AI Agent，目前只有官方稿，传播证据弱，建议观望"
- ❌ "今天最强机会之一"（夸大）
- ❌ "130 个工程师被替代"（单一案例未核实）

### 9.4 示例四：纯技术 benchmark（降级至 B）

```json
{
  "content_category": "technical_context",
  "mention_count": 1,
  "hot_score": 120
}
```

**判断**：技术 benchmark 论文，无应用场景，触发技术内容限制。

**结果**：最高只能评 B，写成"技术趋势观察"。

## 10. 注意事项和风险边界
